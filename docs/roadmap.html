Project plan

Make Vessel trustworthy before making it bigger.

As of May 28, 2026, Vessel already behaves like a real single-node control plane: it deploys curated apps, imports running containers, manages nginx, exposes a web UI, and ships a remote YAML catalog. The next step should harden execution so every later feature sits on a safer base.

Current state

This summary is grounded in the codebase, not in aspirational docs.

Control plane already exists

internal/server/routes.go exposes auth, RBAC, deployments, Docker import, compose stacks, settings, self-update, system helpers, and a full nginx management surface.

Catalog architecture is solid

internal/registry/template.go loads embedded YAML, local overrides, and a remote GitHub Pages catalog, which is the right shape for low-friction template growth.

Frontend is functional but concentrated

internal/server/ui.go contains a large inline SPA. That keeps shipping simple, but it slows iteration, review, and UI-specific testing.

Deployment behavior is synchronous

internal/deployment/engine.go performs pull, up, proxy configuration, and certificate work directly in request-driven flows with coarse status updates like deploying and error.

Operational truth needs cleanup

The README still references Caddy in places, while the active implementation clearly centers on nginx management. That mismatch will confuse operators and contributors.

Quality baseline is usable

Server, store, registry, nginx, and deployment packages already have tests. That is enough to support a focused hardening milestone without first rebuilding the project structure.

Recommended next step

Build an operations layer for long-running actions.

Scope the next milestone around reliable execution

Introduce a persisted operation model for deploy, update, remove, import, bootstrap, nginx actions, and self-update. Treat each action as a tracked workflow with explicit steps, logs, timestamps, actor identity, and final outcome.

  • Add an operations table plus step records so the backend can report more than a single deployment status string.
  • Move long-running work out of fragile request-response handling into background execution with polling or streaming endpoints.
  • Wrap deployment steps with cleanup hooks so failed first-time deploys can remove partial compose files, revert DB state, or clearly mark what remains.
  • Expose operation history in the UI so users can answer: what ran, who started it, where it failed, and whether retry is safe.
  • Write failure-path tests for pull errors, compose up failures, nginx certificate failures, and cleanup behavior.

Why this before more features

  • Vessel is already broad enough. The risk is operator trust, not lack of endpoints.
  • Every future feature, including backups, multi-user audit, and remote execution, depends on durable operation tracking.
  • The code already has the primitives. What is missing is orchestration, visibility, and safe failure handling.

Roadmap

Ordered by leverage, not by novelty.

Phase 1 Trust the control plane

Ship the operations layer, tighten rollback and cleanup rules, add step-level logs, and align docs with the real nginx-based runtime. Goal: make deploy and update actions explainable and safe to retry.

Phase 2 Make state portable

Add backup and restore for the SQLite database, deployment directories, nginx config, and local templates. Goal: let operators move or recover a server without reconstructing it manually.

Phase 3 Unpack the frontend

Extract the inline SPA out of internal/server/ui.go into versioned assets or a small frontend workspace, then add UI tests around auth, deploy flows, and operation history. Goal: speed iteration without losing single-binary shipping.

Phase 4 Expand product depth

Improve template validation, add release notes and update channels, add audit-grade user activity, and consider multi-host or remote-agent work only after single-node reliability is boringly good.

Milestones

Concrete checkpoints for delivery.

M1 Operation records exist

Backend stores operation state, step list, initiator, timestamps, and output pointers. API can list and inspect operations for a deployment or system action.

M2 Deployments fail cleanly

Failed deploys leave either a fully running app or an explainable failed operation with consistent disk and DB state.

M3 Recovery story is real

Backups can be created, validated, and restored on a fresh machine using one documented path.

M4 Frontend is maintainable

The UI is extracted from the giant inline string, and the release process still produces a simple artifact story.

M5 Docs match implementation

README, install flow, and GitHub Pages docs describe the real stack, actual roles, and the new operations workflow.

M6 Expansion work starts from strength

Only after M1 through M5 should the project spend time on broader catalog growth, deeper team features, or multi-host ambitions.

Execution order

A practical sequence for implementation.

Priority Work item Outcome
Now Design the operation schema and backend interfaces. One place to model deploy, update, remove, import, nginx, bootstrap, and self-update actions.
Now Instrument deployment.Engine and nginx actions with named steps and cleanup policies. Failure becomes diagnosable and retryable instead of opaque.
Next Add operation history views and live progress in the embedded UI. Operators can see in-progress and past work without reading backend logs.
Next Align README and docs with nginx, current APIs, and the real product surface. Docs stop fighting the codebase.
Later Build backup/restore and frontend extraction on top of the operations foundation. New features reuse the same execution and reporting model.
